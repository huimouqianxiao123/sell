# 秒杀模块深度优化方案

## 一、当前架构分析

### 现有机制
✅ **已实现的功能：**
- Redis Lua 原子扣减库存
- 本地消息表 + 定时补偿
- 三级幂等控制
- 死信队列处理
- 库存对账补偿

### 存在的问题
❌ **需要优化的地方：**
1. 消息确认机制不够完善
2. 死信队列处理被动
3. 失败回滚策略单一
4. 缺少监控和告警

## 二、优化方案详解

### 2.1 消息确认机制优化

#### 问题
当前流程：
```
消费成功 → 更新 MySQL 状态 → 可能失败导致状态不一致
```

**风险：** 如果 `confirmConsumeSuccess()` 失败，消息会被重复消费

#### 优化方案

**方案一：两阶段确认（推荐）**
```java
// 第一阶段：预确认（记录消费中状态）
rocketMQMessageService.preConfirmConsume(messageId);

// 第二阶段：最终确认（消费成功后）
rocketMQMessageService.finalConfirmConsume(messageId);
```

**方案二：消费日志追踪**
```java
// 记录消费开始时间
long startTime = System.currentTimeMillis();

// 消费成功后记录详细日志
log.info("消费成功：messageId={}, 耗时={}ms, 订单号={}", 
    messageId, System.currentTimeMillis() - startTime, orderNo);

// 更新 MySQL 时增加版本号检查
updateConsumeStatusWithVersion(messageId, status, version);
```

**实现代码：**
```java
/**
 * 预确认消费（记录消费开始时间）
 */
public void preConfirmConsume(String messageId) {
    String key = "seckill:consuming:" + messageId;
    String value = String.valueOf(System.currentTimeMillis());
    stringRedisTemplate.opsForValue().set(key, value, 30, TimeUnit.MINUTES);
}

/**
 * 最终确认消费（带版本号控制）
 */
public void finalConfirmConsume(String messageId, long expectedVersion) {
    // 使用乐观锁更新，防止并发覆盖
    int affected = seckillMessageMapper.updateConsumeStatusWithVersion(
        messageId, 
        SeckillMessage.CONSUME_STATUS_SUCCESS,
        expectedVersion
    );
    if (affected == 0) {
        log.warn("消费确认失败，版本不匹配或消息不存在：{}", messageId);
        throw new RuntimeException("消费确认失败");
    }
}
```

### 2.2 死信队列优化

#### 问题
- 死信消息 1 小时才被扫描一次
- 缺少死信原因分析
- 依赖人工介入

#### 优化方案

**方案一：死信消息自动分类处理**

```java
/**
 * 死信原因分类：
 * 1. TEMPORARY - 临时故障（网络抖动、数据库锁）→ 自动重试
 * 2. BUSINESS - 业务异常（库存不足、重复购买）→ 跳过并告警
 * 3. SYSTEM - 系统异常（Redis 宕机、MQ 故障）→ 延迟重试
 */
public enum DeadLetterReason {
    TEMPORARY,  // 临时故障
    BUSINESS,   // 业务异常
    SYSTEM      // 系统异常
}
```

**方案二：死信消息自动恢复**

```java
/**
 * 死信消息自动恢复策略：
 * - 临时故障：立即重试（最多 3 次）
 * - 系统故障：延迟 5 分钟后重试
 * - 业务异常：记录日志，发送告警，不重试
 */
@Scheduled(cron = "0 */10 * * * ?")  // 每 10 分钟执行一次
public void autoRecoverDeadLetterMessages() {
    List<SeckillMessage> deadLetters = seckillMessageMapper.findDeadLetterMessages(
        SeckillMessage.MAX_SEND_RETRY,
        SeckillMessage.MAX_CONSUME_RETRY,
        50
    );
    
    for (SeckillMessage message : deadLetters) {
        DeadLetterReason reason = analyzeDeadLetterReason(message);
        
        switch (reason) {
            case TEMPORARY:
                // 立即重试
                retryDeadLetterMessage(message);
                break;
            case SYSTEM:
                // 延迟重试
                scheduleDelayedRetry(message, 5, TimeUnit.MINUTES);
                break;
            case BUSINESS:
                // 发送告警，不重试
                alertService.alertDeadLetterBusinessError(message);
                break;
        }
    }
}

/**
 * 分析死信原因
 */
private DeadLetterReason analyzeDeadLetterReason(SeckillMessage message) {
    String failReason = message.getFailReason();
    if (failReason == null) {
        return DeadLetterReason.SYSTEM;
    }
    
    // 业务异常特征
    if (failReason.contains("重复购买") || 
        failReason.contains("库存不足") ||
        failReason.contains("商品不存在")) {
        return DeadLetterReason.BUSINESS;
    }
    
    // 临时故障特征
    if (failReason.contains("timeout") ||
        failReason.contains("Connection") ||
        failReason.contains("Lock")) {
        return DeadLetterReason.TEMPORARY;
    }
    
    // 默认为系统异常
    return DeadLetterReason.SYSTEM;
}
```

### 2.3 失败回滚优化

#### 问题
- 只在 MySQL 持久化失败时回滚
- 消费失败后库存长期占用
- 缺少分级回滚策略

#### 优化方案

**方案一：分级回滚策略**

```java
/**
 * 分级回滚策略：
 * 
 * Level 1: 轻度失败（网络抖动）
 * - 保持 Redis 库存占用
 * - 触发 MQ 重试
 * 
 * Level 2: 中度失败（数据库异常）
 * - 保持 Redis 库存占用
 * - 记录失败日志
 * - 等待定时任务处理
 * 
 * Level 3: 重度失败（业务异常）
 * - 立即回滚 Redis 库存
 * - 清理幂等标记
 * - 发送告警
 */
public enum RollbackLevel {
    LEVEL1_KEEP_STOCK,      // 保持库存，重试
    LEVEL2_WAIT_COMPENSATE, // 等待补偿
    LEVEL3_ROLLBACK_NOW     // 立即回滚
}

/**
 * 根据失败类型决定回滚策略
 */
public RollbackLevel determineRollbackLevel(Exception e, int retryCount) {
    // 业务异常 → 立即回滚
    if (e instanceof BusinessException) {
        return RollbackLevel.LEVEL3_ROLLBACK_NOW;
    }
    
    // 系统异常且重试次数少 → 保持库存
    if (retryCount < 2) {
        return RollbackLevel.LEVEL1_KEEP_STOCK;
    }
    
    // 重试次数多 → 等待补偿
    return RollbackLevel.LEVEL2_WAIT_COMPENSATE;
}
```

**方案二：超时自动回滚**

```java
/**
 * 超时自动回滚任务
 * 每 5 分钟扫描一次，回滚超过 30 分钟未消费的 Redis 库存
 */
@Scheduled(cron = "0 */5 * * * ?")
public void timeoutAutoRollback() {
    // 扫描超过 30 分钟未消费的消息
    List<SeckillMessage> timeoutMessages = seckillMessageMapper.findTimeoutMessages(30, 100);
    
    for (SeckillMessage message : timeoutMessages) {
        log.warn("消息超时未消费，开始回滚 Redis 库存：{}", message.getMessageId());
        
        // 原子回滚 Redis 库存
        boolean success = seckillRollbackService.rollback(
            message.getUserId(), 
            message.getSeckillProductId()
        );
        
        if (success) {
            // 更新消息状态为已取消
            seckillMessageMapper.updateStatus(
                message.getId(), 
                SeckillMessage.STATUS_CANCELLED
            );
            log.info("超时回滚成功：{}", message.getMessageId());
        } else {
            log.error("超时回滚失败：{}", message.getMessageId());
        }
    }
}
```

### 2.4 监控和告警优化

#### 新增监控指标

```java
/**
 * 秒杀监控指标
 */
@Component
public class SeckillMetrics {
    
    @Resource
    private MeterRegistry meterRegistry;
    
    // 消息积压数
    private final AtomicInteger messageBacklog = new AtomicInteger(0);
    
    // 消费延迟（毫秒）
    private final AtomicLong consumeDelay = new AtomicLong(0);
    
    // 失败率
    private final AtomicDouble failureRate = new AtomicDouble(0.0);
    
    // 回滚次数
    private final AtomicInteger rollbackCount = new AtomicInteger(0);
    
    @PostConstruct
    public void init() {
        // 注册 Prometheus 指标
        meterRegistry.gauge("seckill.message.backlog", messageBacklog);
        meterRegistry.gauge("seckill.consume.delay", consumeDelay);
        meterRegistry.gauge("seckill.failure.rate", failureRate);
        meterRegistry.counter("seckill.rollback.count", rollbackCount);
    }
    
    /**
     * 更新监控指标
     */
    public void updateMetrics(int backlog, long delay, double failureRate) {
        this.messageBacklog.set(backlog);
        this.consumeDelay.set(delay);
        this.failureRate.set(failureRate);
        
        // 触发告警
        if (backlog > 1000) {
            alertService.alertMessageBacklog(backlog);
        }
        if (delay > 10000) { // 10 秒
            alertService.alertConsumeDelay(delay);
        }
        if (failureRate > 0.05) { // 5%
            alertService.alertHighFailureRate(failureRate);
        }
    }
}
```

#### 告警规则

```yaml

          summary: "秒杀失败率超过 5%"
```

## 五、风险评估

### 潜在风险
1. **两阶段确认增加复杂度** → 充分测试
2. **自动回滚可能误判** → 保留人工干预入口
3. **监控指标增加系统开销** → 控制采样频率

### 回滚方案
- 保留原有代码分支
- 新增功能通过配置开关控制
- 出现问题时快速回退
