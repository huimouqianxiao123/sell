# 秒杀模块优化报告

## 优化概述

本次检查对秒杀模块进行了全面审查，发现并修复了 6 个潜在问题，进一步提升了系统的可靠性和性能。

## 已完成的优化

### 1. ✅ 修复 Lua 脚本的过期时间逻辑问题

**问题描述：**
原代码只在 Key 不存在时才设置过期时间，但如果 Key 的剩余 TTL 很短（如 1 秒），不会重新设置，可能导致 Key 在秒杀过程中突然过期。

**修复方案：**
修改 `src/main/resources/lua/seckill.lua`，增加对剩余 TTL 的检查：

```lua
-- 修改前
if redis.call('ttl', userListKey) == -1 then
    redis.call('expire', userListKey, expireTime)
end

-- 修改后
local currentTTL = redis.call('ttl', userListKey)
if currentTTL == -1 or currentTTL < expireTime then
    redis.call('expire', userListKey, expireTime)
end
```

**影响：** 防止 Redis Key 提前过期导致的库存丢失问题。

### 2. ✅ 优化 RocketMQMessageService 的异常处理

**问题描述：**
原代码捕获所有 Exception 但不区分业务异常和系统异常，可能掩盖严重的系统问题（如数据库连接断开、Redis 异常等）。

**修复方案：**
在 `RocketMQMessageService.java` 中增加 `isSystemException()` 方法，识别系统级异常并记录更详细的日志：

```java
private boolean isSystemException(Exception e) {
    String message = e.getMessage();
    if (message == null) {
        return false;
    }
    // 数据库连接异常
    if (message.contains("Connection") || message.contains("database") || message.contains("MySQL")) {
        return true;
    }
    // Redis 异常
    if (message.contains("Redis") || message.contains("Jedis") || message.contains("redis")) {
        return true;
    }
    // MQ 连接异常
    if (message.contains("RocketMQ") || message.contains("broker") || message.contains("nameserver")) {
        return true;
    }
    // 网络异常
    if (message.contains("timeout") || message.contains("connect") || message.contains("network")) {
        return true;
    }
    return false;
}
```

**影响：** 系统级异常会立即触发告警日志，便于快速发现和解决问题。

### 3. ✅ 修复订单表的竞态条件问题

**问题描述：**
在极端并发场景下，两个相同的消息可能同时通过订单存在性检查，导致创建重复订单。

**修复方案：**
创建数据库索引优化脚本 `src/main/resources/sql/seckill_indexes.sql`，添加唯一索引：

```sql
-- 防止用户重复购买同一商品的唯一索引
ALTER TABLE orders 
ADD UNIQUE INDEX uk_user_product_order_type_status (
    user_id, 
    product_id, 
    order_type, 
    status
) COMMENT '防止用户重复购买同一商品的唯一索引';
```

**影响：** 数据库层面的最后一道防线，确保数据一致性。

**其他索引优化：**
- 为消息表添加多个复合索引，提升定时任务查询性能
- 为订单表添加索引，优化秒杀订单查询

### 4. ✅ 优化库存对账任务的批量处理

**问题描述：**
原代码一次性取出 pending 队列中所有条目，如果队列非常大（如几万个用户），会导致 Redis 阻塞。

**修复方案：**
在 `SeckillStockReconciliationTask.java` 中实现分批处理：

```java
private static final int BATCH_SIZE = 100;

// 分批获取 pending 队列中的条目
int start = 0;
while (true) {
    List<String> batchUserIds = stringRedisTemplate.opsForList().range(
            pendingKey, start, start + BATCH_SIZE - 1);
    if (batchUserIds == null || batchUserIds.isEmpty()) {
        break;
    }
    
    // 处理批次数据...
    
    start += BATCH_SIZE;
    
    // 每批处理后短暂休眠，避免对 Redis 造成过大压力
    try {
        Thread.sleep(10);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        break;
    }
}
```

**影响：** 避免 Redis 阻塞，提升系统稳定性和性能。

### 5. ✅ 修复消息补偿任务的告警机制

**问题描述：**
原代码在消息重试失败时只是记录日志，没有告警机制，可能导致问题消息长期得不到处理。

**修复方案：**
在 `SeckillMessageCompensationTask.java` 中增加告警逻辑：

```java
// 检查是否需要告警（重试次数接近上限）
if (message.getRetryCount() >= SeckillMessage.MAX_SEND_RETRY - 1) {
    needAlertCount++;
    alertService.alertStockAnomaly(
            message.getSeckillProductId(),
            message.getUserId(),
            String.format("消息发送重试即将耗尽 (重试次数：%d/%d, 消息 ID: %s)", 
                    message.getRetryCount(), 
                    SeckillMessage.MAX_SEND_RETRY,
                    message.getMessageId()));
}
```

**影响：** 当消息重试次数接近上限时自动告警，便于人工及时介入处理。

### 6. ✅ 优化死信消费者的幂等保护

**问题描述：**
死信消息可能被重复消费多次，虽然 `processOrder` 内部有订单重复检查，但缺少 Redis 层面的幂等保护。

**修复方案：**
在 `SeckillDeadLetterConsumer.java` 中增加 Redis 幂等检查：

```java
private static final String DLQ_IDEMPOTENT_KEY_PREFIX = "seckill:dlq:idempotent:";

// 增加 Redis 幂等检查，防止死信消息重复消费
String dlqIdempotentKey = DLQ_IDEMPOTENT_KEY_PREFIX + messageId;
Boolean isFirstVisit = stringRedisTemplate.opsForValue().setIfAbsent(
        dlqIdempotentKey, "1", IDEMPOTENT_EXPIRE_HOURS, TimeUnit.HOURS);

if (!Boolean.TRUE.equals(isFirstVisit)) {
    log.warn("【死信消费】消息已处理过，跳过重复消费，消息 ID: {}", messageId);
    return;
}
```

**影响：** 防止死信消息重复消费，提升系统可靠性。
