# 秒杀模块技术文档

## 一、模块概述

本秒杀模块是一个面向高并发场景的电商秒杀系统，采用 **Redis + RocketMQ + MySQL** 三层架构设计，通过异步化、幂等控制、本地消息表、定时补偿等机制保障数据一致性和系统可靠性。

### 核心设计理念

> **以 Redis 为前线挡流量，以 MQ 为中军做削峰，以 MySQL 为后盾保数据。**

## 四、核心代码深度分析

### 4.1 Lua 脚本 — 秒杀的"原子弹"

**文件：** `resources/lua/seckill.lua`

```lua
-- 1. 校验用户是否重复购买
if redis.call('sismember', userListKey, userId) == 1 then
    return -1  -- 重复购买
end

-- 2. 校验库存
local stock = tonumber(redis.call('get', stockKey))
if stock == nil or stock <= 0 then
    return 0   -- 库存不足
end

-- 3. 扣减库存 & 记录用户 & 写入待处理队列（原子操作）
redis.call('decr', stockKey)
redis.call('sadd', userListKey, userId)
redis.call('lpush', pendingKey, userId)

return 1       -- 秒杀成功
```

**设计要点：**
- 整个脚本在 Redis 单线程中原子执行，**不存在并发竞争**
- `pending` 队列是对账补偿的关键——即使后续 MQ/MySQL 全部失败，也能通过 pending 队列追踪到"库存被扣但订单未生成"的异常情况
- 返回值语义清晰：`-1`=重复、`0`=售罄、`1`=成功

### 4.2 可靠消息发送 — 本地消息表模式

**文件：** `service/Imp/RocketMQMessageService.java`

核心方法 `reliableSend()` 的执行流程：

```
Redis幂等检查(SETNX) → 生成消息ID → 同步写MySQL → 异步发MQ → 回调更新状态
```

**关键设计决策——MySQL 同步写入：**

```java
// 同步持久化到MySQL本地消息表（关键改动：从异步改为同步）
// 确保MySQL中有记录后再发MQ，这样即使MQ发送失败，补偿任务也能兜底
seckillMessageMapper.insert(seckillMessage);
```

这是整个可靠性保障的核心：**先持久化，再发消息**。MySQL 写入成功意味着这条秒杀请求"不会丢"——即使 MQ 发送失败，定时任务扫描 MySQL 后也能重发。

**失败处理链：**

```
MySQL写入失败 → 清理Redis幂等标记 → 抛异常 → 调用方回滚Redis库存 → 用户可重试
MQ发送失败  → 回调中更新MySQL状态为失败 → 定时任务30秒后自动重发
```

### 4.3 MQ 消费者 — 事务性订单创建

**文件：** `consumer/MiaoShaOrderConsumer.java`

`processOrder()` 方法使用 `@Transactional` 包裹，确保以下操作原子性：

```java
@Transactional(rollbackFor = Exception.class)
public String processOrder(Long userId, Long seckillProductId, Long productId) {
    // 1. MySQL库存扣减（stock > 0 条件，最后防线）
    int affectedRows = seckillProductMapper.decreaseStockWithoutVersion(seckillProductId);
    // 2. 创建 Orders 订单
    orderMapper.insert(order);
    // 3. 创建 OrderItem 订单项
    orderItemMapper.insert(orderItem);
    return orderNo;
}
```

**MySQL 库存扣减 SQL：**

```sql
UPDATE seckill_product
SET seckill_stock = seckill_stock - 1, version = version + 1
WHERE id = #{id} AND seckill_stock > 0
```

`AND seckill_stock > 0` 是防超卖的最后一道关卡——即使 Redis 层出现异常导致多扣，MySQL 层也能兜住。

### 4.4 三级幂等体系

**文件：** `service/Imp/SeckillIdempotentService.java`

| 层级 | Key 前缀 | 有效期 | 作用 |
|------|---------|--------|------|
| **请求级** | `seckill:idempotent:{userId}:{productId}` | 60秒 | 防用户短时间内重复点击 |
| **消息级** | `seckill:message:{userId}:{productId}` | 300秒 | 防同一用户同一商品的消息重复发送 |
| **消费级** | `seckill:consume:{messageId}` | 3600秒 | 防 MQ 消息重复消费 |

三级幂等形成层层过滤，从用户请求到 MQ 消费的每一环节都有去重保护。

### 4.5 定时任务矩阵

#### 4.5.1 秒杀预热任务（MiaoShaTask）

| 子任务 | 频率 | 分布式锁 | 核心逻辑 |
|--------|------|---------|---------|
| `prepareSeckillProduct` | 每分钟 | UUID安全锁 | 预热未来5分钟内商品 + 恢复缓存丢失的进行中商品 |
| `updateSeckillStatus` | 每分钟 | 无 | 状态机流转：0(未开始)→1(进行中)→2(已结束)/3(已售罄) |
| `syncStockToDatabase` | 每5分钟 | 无 | Redis库存→MySQL（只减不增策略） |
| `cleanExpiredSeckillCache` | 每小时 | 无 | 清理结束超1小时的缓存 |

**预热库存计算公式：**

```java
实际剩余库存 = 初始秒杀库存
             - seckill_message 表已售数量（status=1 且 consume_status=1）
             - orders 表普通订单已售数量（order_type=1）
```

**预热使用版本号 Lua 脚本（seckill_warmup.lua）防止旧数据覆盖新数据：**

```lua
local currentVersion = redis.call('get', versionKey)
if currentVersion and tonumber(currentVersion) >= tonumber(version) then
    return 0  -- 已有更新版本，跳过
end
redis.call('set', stockKey, stock, 'EX', expireSeconds)
redis.call('set', versionKey, version, 'EX', expireSeconds)
return 1
```

#### 4.5.2 对账补偿任务（SeckillStockReconciliationTask）

每2分钟扫描 Redis pending 队列，处理"扣了库存但订单未生成"的异常条目：

```
pending 中有用户ID
    ├─ MySQL消息已消费成功 → 清理pending（正常情况，只是pending没被及时清理）
    ├─ MySQL消息存在但未消费 → 清理pending（交由消息补偿任务处理）
    └─ MySQL无记录 → 补发消息/回滚Redis库存（极端异常情况）
```

#### 4.5.3 消息补偿任务（SeckillMessageCompensationTask）

| 子任务 | 频率 | 处理目标 |
|--------|------|---------|
| `compensateSendFailedMessages` | 每30秒 | 发送失败/待发送的消息（重试≤5次） |
| `compensateConsumeFailedMessages` | 每分钟 | 消费失败的消息（重试≤3次，先查订单是否已存在） |
| `checkTimeoutMessages` | 每5分钟 | 发送成功但超10分钟未消费的消息 |
| `handleDeadLetterMessages` | 每小时 | 超过最大重试次数的死信消息（告警） |
| `statisticsMessages` | 每小时 | 消息处理情况统计日志 |

## 七、优势分析

### 7.1 防超卖能力强

系统在 **三个层面** 防止超卖：

1. **Redis Lua 原子操作**：检查+扣减在单线程中原子执行，并发安全
2. **MySQL 条件更新**：`WHERE seckill_stock > 0`，即使 Redis 异常，MySQL 也不会超扣
3. **版本号机制**：预热时用 Lua 脚本做版本检查，防止旧库存数据覆盖新数据

### 7.2 消息可靠性高

采用经典的 **本地消息表** 模式，形成完整的消息生命周期管理：

```
创建(status=0) → 发送成功(status=1) → 消费成功(consume_status=1)
                  ↓ 失败               ↓ 失败
            发送失败(status=2)    消费失败(consume_status=2)
                  ↓                    ↓
            定时补偿重发           定时补偿重发
                  ↓                    ↓
           超过5次 → 死信告警    超过3次 → 死信告警
```

### 7.3 异常恢复能力强

四道防线形成**纵深防御**：

| 防线 | 场景 | 恢复方式 |
|------|------|---------|
| 第一道 | MQ 发送失败 | `sendToMq` 中立即回滚 Redis 库存 |
| 第二道 | MySQL 持久化失败 | 清理幂等标记 → 回滚 Redis → 用户可重试 |
| 第三道 | 回滚也失败 | pending 队列对账任务（每2分钟）发现并修复 |
| 第四道 | 消息发送成功但消费失败 | 消息补偿任务（每30秒~每小时）重发 |

### 7.4 高并发吞吐

- **请求拦截在 Redis**：绝大多数请求被 Lua 脚本在毫秒级拒绝（重复/售罄），不穿透到 MySQL
- **异步削峰**：通过 RocketMQ 将瞬时流量平滑化，消费者按自己的节奏处理
- **异步 IO 线程池**：消息状态更新、缓存预热等 IO 操作使用独立线程池，不阻塞主流程

### 7.5 可观测性好

- 完整的结构化日志（所有关键操作都有 `log.info`/`log.warn`/`log.error`）
- 消息统计任务每小时输出处理情况
- 死信告警机制（预留了邮件/短信/钉钉扩展点）
- 压测统计接口 `getSeckillOrderCount` 可实时查看各维度数据

### 7.6 预热机制完善

三个入口触发预热，确保库存缓存不会缺失：

1. **定时任务预热**：每分钟扫描，提前5分钟预热即将开始的商品
2. **添加商品预热**：事务提交后异步预热，商品上线即可秒杀
3. **兜底预热**：秒杀请求发现缓存缺失时异步触发，返回"系统繁忙"

## 八、不足与改进建议

### 8.1 分布式锁实现不统一

**问题：** 系统中存在两种分布式锁实现方式：

- `MiaoShaTask.prepareSeckillProduct`：使用 UUID + Lua 脚本安全释放（正确做法）
- `RecoveryStockTask`、`SeckillStockReconciliationTask` 等：使用简单的 `setIfAbsent` + `delete`（存在锁被误删风险）

**分析：** 简单 `delete` 方式的风险场景：任务 A 获取锁后执行时间超过锁的 TTL，锁自动过期，任务 B 获取锁，此时任务 A 执行完毕调用 `delete` 误删了任务 B 的锁。

**建议：** 统一使用 UUID + Lua 脚本释放锁的模式，或引入 Redisson 框架统一管理分布式锁。

### 8.2 `@Transactional` 在非 `public` 方法上的隐患

**问题：** `MiaoShaOrderConsumer.processOrder()` 是 `public` 方法但通过 `this.processOrder()` 调用，Spring AOP 代理不会生效，`@Transactional` 可能失效。

**分析：** 由于 `processOrder` 是在同一个类（`MiaoShaOrderConsumer`）中通过 `this` 直接调用的，Spring 的 CGLIB 代理无法拦截这个调用，导致事务注解实际上不生效。如果 `orderMapper.insert` 成功但 `orderItemMapper.insert` 失败，订单数据和订单项数据将不一致。

**建议：** 将 `processOrder` 提取到独立的 Service 类中，通过注入方式调用以使事务代理生效；或使用 `TransactionTemplate` 编程式事务。

### 8.3 `findActiveProducts` SQL 条件与实际业务不匹配

**问题：** `SeckillProductMapper.findActiveProducts()` 只查询 `status = 0` 的商品：

```sql
SELECT * FROM seckill_product
WHERE status = 0 AND (end_time IS NULL OR end_time > DATE_SUB(NOW(), INTERVAL 24 HOUR))
```

**分析：** 但对账补偿任务需要处理的是"进行中"（status=1）甚至"已售罄"（status=3）的商品的 pending 队列。status=0 是"未开始"的商品，此时不应有 pending 数据。这意味着对账任务可能扫描不到真正需要处理的商品。

### 8.4 pending 队列中 userId 重复问题

**问题：** Lua 脚本中使用 `LPUSH` 将 userId 写入 pending 队列（List 类型），但同一个 userId 可能因为回滚+重试出现多条记录。

**分析：** `LREM pendingKey 1 userId` 每次只移除一个匹配项，如果有多个相同 userId，清理不完整。虽然不影响正确性（对账任务会再次检查），但会造成不必要的重复扫描。

### 8.5 缺少限流/熔断机制

**问题：** 当前系统在 Controller 层没有任何限流措施。

**分析：** 虽然 Redis Lua 脚本执行很快，但在极端高并发下：
- 大量请求同时涌入仍然会消耗网络带宽和 Redis 连接
- 如果 Redis 故障，所有请求会直接超时

**建议：**
- 接入 Sentinel 或 Gateway 做请求级限流（如令牌桶、滑动窗口）
- 添加本地缓存标记（如 `HashMap` 标记商品是否已售罄），在进入 Redis 之前做第一层拦截
- 增加 Redis 熔断机制，Redis 不可用时快速失败而非阻塞

### 8.6 库存同步"只减不增"策略的潜在问题

**问题：** `MiaoShaTask.syncStockToDatabase` 采用"只减不增"策略——Redis 库存 < DB 库存时才同步。

**分析：** 当消费者处理失败回滚 Redis 库存（INCR）后，Redis 库存可能暂时大于 DB 库存。此时同步任务跳过不同步是合理的。但如果回滚后用户成功重新秒杀，新的扣减不一定能及时反映到 MySQL。好在最终数据一致性由 MQ 消费者的 `decreaseStockWithoutVersion` 保证，同步任务只是辅助对齐。

### 8.7 缺少单元测试和集成测试

**问题：** 核心秒杀逻辑缺少自动化测试覆盖。

**建议：**
- 为 Lua 脚本编写单元测试（使用嵌入式 Redis）
- 为 `reliableSend` 的各种失败路径编写测试
- 为 `processOrder` 的事务边界编写集成测试
- 编写高并发压测用例验证防超卖能力

### 8.8 告警通知未落地

**问题：** `SeckillMessageCompensationTask.handleDeadLetterMessages` 中有 TODO 注释但未实现告警通知。

```java
// TODO: 这里可以添加告警通知逻辑
// 1. 发送邮件告警
// 2. 发送短信告警
// 3. 接入告警平台（如钉钉、企业微信）
```

**分析：** 死信消息意味着有用户的秒杀库存被扣但订单未生成，且自动恢复已达上限。仅靠日志难以及时发现，生产环境必须有主动告警。

### 8.9 消息格式兼容带来的复杂性

**问题：** `MiaoShaOrderConsumer.onMessage` 需要兼容两种消息格式：

```
新格式：messageId:userId:seckillProductId[:productId]
旧格式：userId:seckillProductId
```

**分析：** 使用 `:` 分隔的纯文本格式，解析依赖 `parts.length` 判断，脆弱且不易扩展。

**建议：** 使用 JSON 格式传递消息，字段明确、易扩展、不易出错。旧格式可在灰度完成后移除。

### 8.10 线程安全隐患

**问题：** `warmUpSeckillStock` 方法中的重试使用 `Thread.sleep`，如果在异步线程池中执行，阻塞线程池线程。

**分析：** 当大量商品同时需要预热且都失败重试时，线程池可能被阻塞的预热任务占满，影响其他异步任务的执行。

**建议：** 使用 `ScheduledExecutorService.schedule` 替代 `Thread.sleep`，实现非阻塞的延迟重试。

## 十、总结

### 整体评价

这是一个**设计严谨、考虑周全**的秒杀系统实现。核心亮点包括：

1. **三层防护架构**（Redis→MQ→MySQL）层层递进，每一层都有独立的防超卖机制
2. **本地消息表 + 四道补偿防线**，在分布式系统中最大程度保证了数据最终一致性
3. **pending 队列对账机制**，解决了"Redis 扣了库存但后续流程全部中断"这一极端场景
4. **版本号预热机制**，用 Lua 脚本保证库存预热的原子性和幂等性
5. **三级幂等体系**，从请求到消费全链路防重复

### 需要重点关注

1. `processOrder` 的事务代理失效问题（优先级最高）
2. `findActiveProducts` 的 SQL 条件应包含 status=1 的商品
3. 分布式锁的实现需要统一
4. 生产环境必须补齐告警通知
5. 建议为核心流程补充自动化测试
