## 一、消息确认机制优化 - 两阶段确认

### 1.1 问题背景

**原有问题：**
- 消费成功后直接更新 MySQL 状态，如果更新失败会导致状态不一致
- 缺少消费耗时追踪
- 没有版本控制，可能出现并发覆盖

### 1.2 实施方案

**两阶段确认流程：**

```
第一阶段（预确认）：
  消费开始 → preConfirmConsume(messageId)
  ↓
  记录消费开始时间
  ↓
  设置状态为消费中
  
第二阶段（最终确认）：
  消费成功 → finalConfirmConsume(messageId, status)
  ↓
  获取当前版本号
  ↓
  使用乐观锁更新（version = version + 1）
  ↓
  确认成功
```

### 1.3 代码变更

#### 1.3.3 服务层 - RocketMQMessageService.java

**新增方法：**
```java
/**
 * 两阶段确认 - 第一阶段：预确认消费
 */
public boolean preConfirmConsume(String messageId) {
    // 记录消费开始时间
    LocalDateTime now = LocalDateTime.now();
    int affected = seckillMessageMapper.preConfirmConsume(messageId, now);
    return affected > 0;
}

/**
 * 两阶段确认 - 第二阶段：最终确认消费
 */
public boolean finalConfirmConsume(String messageId, int consumeStatus) {
    // 获取当前版本号
    Long currentVersion = seckillMessageMapper.getVersionByMessageId(messageId);
    
    // 使用版本号更新（乐观锁）
    int affected = seckillMessageMapper.updateConsumeStatusWithVersion(
        messageId, consumeStatus, currentVersion);
    
    return affected > 0;
}
```

### 1.4 使用示例

**在消费者中使用：**
```java
public void onMessage(MessageExt messageExt) {
    String messageId = extractMessageId(messageExt);
    
    try {
        // 第一阶段：预确认
        if (!rocketMQMessageService.preConfirmConsume(messageId)) {
            log.warn("预确认失败，跳过");
            return;
        }
        
        // 执行业务逻辑
        String orderNo = seckillOrderService.processOrder(...);
        
        // 第二阶段：最终确认
        if (!rocketMQMessageService.finalConfirmConsume(
                messageId, SeckillMessage.CONSUME_STATUS_SUCCESS)) {
            log.error("最终确认失败，版本冲突");
            throw new RuntimeException("确认失败");
        }
        
        log.info("消费成功，订单号：{}", orderNo);
    } catch (Exception e) {
        // 处理失败
        rocketMQMessageService.finalConfirmConsume(
            messageId, SeckillMessage.CONSUME_STATUS_FAILED);
        throw e;
    }
}
```

### 1.5 优势

✅ **防止重复消费**：版本号控制防止并发覆盖
✅ **可追踪消费耗时**：记录消费开始时间
✅ **提高一致性**：两阶段确认确保状态准确

## 二、死信队列自动分类处理

### 2.1 问题背景

**原有问题：**
- 死信消息 1 小时才被扫描一次，处理延迟高
- 缺少死信原因分析，所有消息一视同仁
- 依赖人工介入，运维成本高

### 2.2 实施方案

**死信原因自动分类：**

```java
public enum DeadLetterReason {
    TEMPORARY,  // 临时故障 → 立即重试
    BUSINESS,   // 业务异常 → 告警不重试
    SYSTEM,     // 系统异常 → 延迟重试
    UNKNOWN     // 未知原因 → 人工介入
}
```

**自动处理策略：**
- **临时故障**：立即重试（网络抖动、数据库锁）
- **系统故障**：延迟 5 分钟后重试（Redis 宕机、MQ 故障）
- **业务异常**：记录日志 + 告警，不重试（重复购买、库存不足）
- **未知原因**：记录日志，等待人工介入

### 2.3 代码变更

#### 2.3.1 枚举类 - DeadLetterReason.java

**新建文件：**

#### 2.3.2 定时任务 - SeckillMessageCompensationTask.java

**新增定时任务（每 10 分钟执行一次）：**
```java
@Scheduled(cron = "0 */10 * * * ?")
public void autoRecoverDeadLetterMessages() {
    // 扫描死信消息
    List<SeckillMessage> deadLetterMessages = seckillMessageMapper.findDeadLetterMessages(...);
    
    for (SeckillMessage message : deadLetterMessages) {
        // 分析死信原因
        DeadLetterReason reason = analyzeDeadLetterReason(message);
        
        switch (reason) {
            case TEMPORARY:
                retryDeadLetterMessage(message);  // 立即重试
                break;
            case SYSTEM:
                scheduleDelayedRetry(message, 5, TimeUnit.MINUTES);  // 延迟重试
                break;
            case BUSINESS:
                alertService.alertStockAnomaly(...);  // 告警
                break;
            default:
                log.error("未知原因，需要人工介入");
        }
    }
}
```

**死信原因分析方法：**
```java
private DeadLetterReason analyzeDeadLetterReason(SeckillMessage message) {
    String failReason = message.getFailReason();
    
    // 业务异常特征
    if (failReason.contains("重复购买") ||
        failReason.contains("库存不足") ||
        failReason.contains("商品不存在")) {
        return DeadLetterReason.BUSINESS;
    }
    
    // 临时故障特征
    if (failReason.contains("timeout") ||
        failReason.contains("Connection") ||
        failReason.contains("Lock")) {
        return DeadLetterReason.TEMPORARY;
    }
    
    // 系统故障特征
    if (failReason.contains("Redis") ||
        failReason.contains("RocketMQ") ||
        failReason.contains("MySQL")) {
        return DeadLetterReason.SYSTEM;
    }
    
    return DeadLetterReason.UNKNOWN;
}
```

**延迟重试实现：**
```java
private void scheduleDelayedRetry(SeckillMessage message, long delay, TimeUnit unit) {
    ioTaskExecutor.execute(() -> {
        unit.sleep(delay);  // 延迟执行
        
        // 重置状态并重试
        seckillMessageMapper.updateConsumeStatus(...);
        rocketMQMessageService.retrySend(message);
    });
}
```

### 2.4 处理流程

```
死信消息扫描（每 10 分钟）
    ↓
分析死信原因
    ↓
┌───────────┬────────────┬────────────┬──────────┐
│ 临时故障   │ 系统故障    │ 业务异常    │ 未知原因  │
└─────┬─────┴──────┬─────┴─────┬──────┴────┬─────┘
      │            │           │           │
  立即重试    延迟 5 分钟重试   发送告警    记录日志
      │            │           │           │
      ↓            ↓           ↓           ↓
  成功/失败    成功/失败    不重试     人工介入
```

### 2.5 优势

✅ **处理延迟降低**：1 小时 → 10 分钟（83% 提升）
✅ **自动分类处理**：智能识别故障类型
✅ **减少人工介入**：临时故障和系统故障自动恢复
✅ **精准告警**：只对业务异常发送告警
